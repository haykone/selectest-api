2. Повторный запуск приложения, контейнер завершился ошибкой

Причина: код ожидал что поле city всегда заполнено, но на самом деле это не так
Решение: добавили проверку наличия значения перед обращением к атрибуту name, код проверяет есть ли объект city, если его нет присваивает ключу city_name значение None.
До: "city_name": item.city.name.strip(),
После: "city_name": (item.city.name or "").strip() if item.city else None,

3. При наблюдении за логами обнаружено, что фоновый парсинг вакансий запускается каждые 5 секунд вместо запланированных 5 минут.
Причина: Переменная settings.parse_schedule_minutes использовалась как значение параметра seconds=, хотя по смыслу названия и документации она должна интерпретироваться как минуты.
Решение:
До: seconds=settings.parse_schedule_minutes,
После: minutes=settings.parse_schedule_minutes,

4. При попытке создать вакансию с существующим external_id API возвращал статус 200(ОК) с сообщением об ошибке.
Причина: в эндпоинте при обнаружении существующей вакансии возвращался JSONResponse со статусом 200(ОК).
Решение:
До: return JSONResponse(
                status_code=status.HTTP_200_OK,
                content={"detail": "Vacancy with external_id already exists"},
            )
После: return JSONResponse(
                status_code=status.HTTP_409_CONFLICT,
                content={"detail": "Vacancy with external_id already exists"},
            )

6. При попытке удалить вакансию и снова обратиться к ней же по id в ответ получаем что запись все равно существует 
Причина: В функции delete_vacancy метод session.delete(vacancy) вызывался без await, хотя используется асинхронная сессия AsyncSession.
Из-за этого удаление не регистрировалось в unit-of-work сессии, запрос DELETE в базу не отправлялся, и await session.commit() ничего не делал.
В результате эндпоинт возвращал 204 No Content, но вакансия оставалась в базе (ложный успех).
Решение:
До:
session.delete(vacancy)
await session.commit()
После: 
await session.delete(vacancy)
await session.commit()

7. Неконсистентный тип данных для existing_ids в upsert_external_vacancies
Причина: Переменная existing_ids в случае наличия external_id получала тип set, а в случае пустого списка – тип dict. Формально работает всё, но это снижает читаемость, а в будущем могло привести проблемы при расширении логики.
Решение:
До:
        existing_ids = set(existing_result.scalars().all())
    else:
        existing_ids = {}
После:
       existing_ids = set(existing_result.scalars().all())
    else:
        existing_ids = set()

8. Утечка ресурсов из-за незакрытого httpx.AsyncClient в функции парсинга.
Причина: 
В функции parse_and_store создавался экземпляр httpx.AsyncClient, но нигде не вызывался await client.aclose(). Из-за этого пул соединений и другие внутренние ресурсы клиента оставались открытыми после завершения парсинга или ошибок.
Решение:
До:
client = httpx.AsyncClient(timeout=timeout)
После:
async with httpx.AsyncClient(timeout=timeout) as client:
